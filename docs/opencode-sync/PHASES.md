# 一期 / 二期边界

## 1. 版本策略

- 一期目标：先交付一个能独立使用的同步工具。
- 二期目标：在不推翻一期架构的前提下补齐后台自动化能力。

## 2. 一期边界

### 2.1 一期必须做

1. 项目初始化与配置生成。
2. 基于 OpenCode CLI 的会话列举、导出、导入。
3. `push`、`pull`、`sync` 基本闭环。
4. session 级去重与增量判断。
5. `run` 模式。
6. 一个最小可用远端驱动。
7. manifest 读写。

### 2.2 一期应该做

1. `status` 查看本地与远端差异。
2. `doctor` 检查本地依赖和配置。
3. 冲突提示与冲突副本保留策略。
4. 本地同步状态文件。

### 2.3 一期明确不做

1. 常驻后台。
2. 文件系统实时监听。
3. 多人协作锁。
4. 云端控制台。
5. 消息级别合并。
6. 大规模团队权限管理。

## 3. 二期边界

### 3.1 二期优先能力

1. `daemon start` / `stop` / `status`。
2. 周期性自动同步。
3. 变更监听与自动触发 push。
4. 冲突通知。
5. 多后端支持。

### 3.2 二期增强能力

1. S3-compatible 存储。
2. 自托管 API。
3. 机器元数据与活跃状态。
4. 更细的同步策略配置。
5. 远端清理和保留策略。

## 4. 架构约束

为了确保二期能平滑叠加，一期必须遵守以下约束：

1. 后台不是核心逻辑，只是调度层。
2. 所有同步能力都必须可由 CLI 单独调用。
3. manifest、session blob、配置格式一旦确定，应尽量保持向后兼容。
4. 存储驱动接口必须独立于 CLI 层。

## 5. 推荐实施顺序

1. OpenCode adapter
2. manifest schema
3. local state
4. push/pull/sync
5. run
6. status/doctor
7. daemon skeleton
