# 产品需求文档 / PRD

## 1. 产品定位

- `opencode-sync` 是一个围绕 OpenCode CLI 的会话同步工具。
- 它的目标不是替代 OpenCode，也不是改写 OpenCode 内部存储，而是基于 OpenCode 已公开的 `session list`、`export`、`import` 能力，提供跨机器、跨环境的会话同步与恢复能力。
- 产品既支持纯工具模式，也支持后续可选的后台模式；一期优先交付纯 CLI 闭环。

## 2. 背景与问题

当前 OpenCode 会话主要存在本机环境中，存在以下痛点：

1. 用户在 A 机器上的 OpenCode 对话，无法自然地在 B 机器继续使用。
2. 即便用户在同一项目中切换设备，也缺少统一的会话云端同步机制。
3. 用户希望尽量保留 OpenCode 原生使用方式，不希望依赖侵入式补丁或修改 OpenCode 内部数据库。
4. 有些用户只需要手动同步工具，有些用户则希望后期能开启更自动的后台同步。

## 3. 产品目标

1. 支持把本地 OpenCode 会话导出并同步到远端存储。
2. 支持另一台机器将这些远端会话拉取并重新导入 OpenCode。
3. 支持同一产品同时覆盖两种使用方式：
   - 无后台模式：手动命令触发。
   - 有后台模式：后续可选的自动同步调度。
4. 不依赖 OpenCode 私有内部实现，优先使用官方 CLI 命令。
5. 首版设计应为未来扩展到 GitHub、S3、API、自托管等后端保留接口。

## 4. 非目标

一期不纳入：

1. 修改 OpenCode 内部数据库或内部文件结构。
2. 实时消息级同步。
3. 多人同时编辑同一会话的复杂合并。
4. Web 管理后台。
5. 团队权限系统。
6. 云端全文检索与分析控制台。

## 5. 目标用户

| 角色 | 主要诉求 |
|------|----------|
| 个人开发者 | 在多台机器间同步自己的 OpenCode 会话 |
| 重度 CLI 用户 | 希望用最少命令完成会话迁移与恢复 |
| 小团队用户 | 后期希望共享远端存储能力，但不一定一开始就部署服务端 |

## 6. 核心场景

### 场景 A：换机继续工作

1. 用户在机器 A 的某个项目中使用 OpenCode 产生会话。
2. 执行 `opencode-sync push` 或 `opencode-sync sync`。
3. 在机器 B 的同项目目录下执行 `opencode-sync pull`。
4. 会话被导入本机 OpenCode，用户可继续在 OpenCode 中查看或续写。

### 场景 B：把 `opencode-sync` 包装成统一入口

1. 用户在项目中执行 `opencode-sync run`。
2. 工具先执行一次 `pull`。
3. 然后启动 `opencode`。
4. OpenCode 退出后自动执行一次 `push`。

### 场景 C：后续增强为后台自动同步

1. 用户启用 daemon 模式。
2. 工具在本地周期性扫描变更。
3. 自动执行 push/pull。
4. 发生冲突时给出提示而不是静默覆盖。

## 7. 产品原则

1. 先稳定，再自动化。
2. 先 session 级同步，再考虑更细粒度。
3. 先兼容 OpenCode 官方能力，再考虑更深集成。
4. 先做一个能独立工作的工具，再加后台模式。
5. 所有远端同步都以可审计、可恢复、可重试为原则。

## 8. 一期范围

一期必须实现：

1. OpenCode 会话扫描。
2. OpenCode 会话导出。
3. OpenCode 会话导入。
4. 本地与远端 manifest 对比。
5. 基于 session 的增量 push。
6. 基于 session 的增量 pull。
7. `run` 包装命令。
8. 最少一种远端存储驱动。

一期推荐优先实现：

1. `github` 驱动。
2. 统一项目标识 `projectKey`。
3. `status`、`doctor`。
4. 本地同步状态缓存。

## 9. 二期范围

二期可实现：

1. daemon 常驻进程。
2. 定时同步。
3. 自动监听本地会话更新。
4. S3 / S3-compatible 存储驱动。
5. 自托管 API 驱动。
6. 更完整的冲突处理。
7. 多机器/多人协作元数据。

## 10. 成功指标

1. 用户可以在两台机器之间完成一次完整的 OpenCode 会话同步与恢复。
2. 常见项目中，执行一次 `push` / `pull` 的操作步骤应控制在 1 到 2 条命令内。
3. 相同 session 未发生变化时，不应重复上传相同内容。
4. 导入后的 session 能在 OpenCode 中正常出现并可继续使用。
5. 用户在纯 CLI 模式下即可完成核心任务，不依赖后台进程。

## 11. 关键风险

1. `opencode import` 对重复导入的处理方式需要验证。
2. 同一 session 在多机修改后，冲突策略需要明确。
3. 不同 OpenCode 版本之间导出格式可能存在变化。
4. 部分用户可能期望实时同步，但一期只提供命令式同步。
